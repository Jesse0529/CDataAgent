# CData Agent — 面试抗压准备

> 系统架构深度提问 20 问。每题含：问题 → 精确答案（符合项目实际，不赘述）。

---

## Q1：为什么用双层 Agent 而不是单 Agent？为什么不用三层？

**单 Agent 问题**：14 个工具平铺时 LLM 决策空间过大，容易选错工具，且数据操作和图表输出混在一起。

**三层问题**：Planner 意图推理增加 1-2s 延迟且对 DeepSeek 推理要求高，实测中 DeepSeek 在规划角色表现不稳定，经常生成不可执行的计划。移除 Planner 后改为 Executor 自主决策 + 意图守卫兜底，效果等价但减少一次 LLM 调用。

**双层优势**：Executor 专注数据操作（4 个工具）、Synthesizer 专注输出（2 个工具），决策空间小，出错率低。数据通过 AnalysisState 的 outputKey → dataJson 传递，不占 LLM 上下文。

---

## Q2：Agent 之间怎么通信？数据怎么流转？

**不通过 LLM 上下文传递**。Executor 工具调用结果通过 `AnalysisState.dataIndex`（outputKey → dataJson）存储。Synthesizer 的 `ChartOutputTool.buildChart` 按 dataRef 参数从 dataIndex 读取数据。

协调信号：Executor 流输出中检测 `##NEEDS_CHART##` 标记时，AgentServiceImpl 触发 Synthesizer 阶段。Synthesizer 的 chart 和 text 结果最后合并为 SSE 响应。

---

## Q3：意图守卫具体怎么工作的？真的能强制 LLM 先声明意图吗？

**流程**：每轮开始 `clearIntent()` 将 `intentCategory=null`。Agent 调用 DataLoadingTool 或 DuckDbQueryTool 时，先执行 `checkIntentGuard()`：
- `null` → 返回 `"请先调用 declareIntent 声明意图"`
- `chitchat/vague` → 返回对应错误
- `analysis` → 放行

**不是代码强制**：守卫返回**结构化 JSON 错误给 LLM**，LLM 根据 System Prompt 指引自我修正。如果 LLM 同时忽略 prompt 和错误（理论上可能），无框架级阻止机制。Prompt 明确写了"这是强制性的第一步"，加上守卫报错构成双重保障，实际效果可靠。

**设计隐患**：`intentCategory` 是普通实例字段，未按 threadId 隔离。单用户串行场景无问题，并发场景有交叉风险。

## Q4：为什么选 DuckDB + H2 组合？为什么不直接用 MySQL？

| 组件 | 角色 | 选型理由 |
|------|------|---------|
| DuckDB | 分析引擎 | 嵌入式列存，Parquet 原生支持，零部署，OLAP 查询性能远超 H2/SQLite |
| H2 | 元数据注册 | Spring Boot 原生支持，零安装，文件级持久化 |

**不选 MySQL**：单用户场景不需独立数据库服务。H2 文件路径在配置中定义，生产环境如需扩展可切换到 MySQL（配置变更即可，MyBatis-Plus 兼容）。

## Q5：文件上传到可查询的完整流程是什么？

```
上传 → 校验（扩展名 + 大小）→ SHA256 去重
  → 命中 → 复用，跳过转换
  → 未命中 → XLSX：EasyExcel 流式读 → 临时 CSV
           → CSV：直接处理
  → DuckDB read_csv_auto（UTF-8→GBK 回退）→ COPY TO Parquet（Snappy）
  → 清理临时文件
  → DESCRIBE 获取 schema（列名/类型/采样值）
  → 写入 H2 data_file 表（含 columnMeta JSON）
  → 标记 status=READY
```

**事务保证**：批量上传时全部转换成功后才删除旧文件、写入新记录，任一步失败回滚新产生的临时 Parquet。

## Q6：图表是 LLM 生成的还是代码生成的？怎么保证格式正确？

**代码生成**：使用 `echarts-java` 库根据工具参数（chartType、title、dimensionField、metricMapping、dataRef）确定性构建 ECharts v5 option JSON。LLM 只负责决定"用什么图表类型、X 轴是什么、指标是什么"，不手写 JSON。

**双保险**：`buildChart` 生成后自动调用 `validateChart` 校验 JSON 结构（检查 title.text、series、xAxis/yAxis 等必填字段，验证 series.data 与 dimensionField 一致性），校验结果返回给 LLM，LLM 可据此修正参数重试。

---

## Q7：多文件 JOIN 怎么实现的？

DuckDB 不支持跨数据库实例查询，但支持在同一连接中注册多个 VIEW。

**实现**：`DuckDbConfig.createConnection(List<FileRef>)` 遍历文件列表，为每个 Parquet 执行 `CREATE OR REPLACE VIEW viewName AS SELECT * FROM 'parquetPath'`。Agent SQL 中通过 `data_<uuid>` 格式的 viewName 引用具体文件，DuckDB 原生支持跨 VIEW 的 JOIN 查询。视图名（viewName）在文件入库时生成，与 H2 data_file 表关联。

## Q8：SQL 注入怎么防护的？

**白名单 + 模式过滤双层防护**：

1. 顶层关键字白名单：仅允许 `SELECT / DESCRIBE / SHOW / EXPLAIN / WITH` 开头
2. 危险模式过滤：禁止多语句（分号分段）、DDL/DML（INSERT/UPDATE/DELETE/DROP/ALTER）、写入操作
3. 自动追加 `LIMIT`：无 LIMIT 的 SELECT 自动追加 `LIMIT maxResultRows`
4. 查询超时：`stmt.setQueryTimeout(30)` 防止慢查询耗尽资源
5. SQL 参数化：使用 `PreparedStatement`，Agent 不拼接 SQL（参数由 Tool 注解定义）

## Q9：拦截器链为什么要两层重试？参数怎么设计的？

**两层原因**：不同工具的错误模式不同。SQL 查询超时或语法错误可重试，Python 沙箱因环境问题失败也可重试。但需要**差异化配置**让各自只处理属于自己的异常类型。

| 参数 | duckdbRetry | sandboxRetry |
|------|-------------|-------------|
| 触发条件 | SQLException | 所有 Exception |
| 重试次数 | 1 次 | 1 次 |
| 初始延迟 | 500ms | 1000ms |
| 退避因子 | 2.0 | 2.0 |
| 抖动 | true（±25%） | true |
| 失败行为 | RETURN_MESSAGE | RETURN_MESSAGE |
| 错误提示 | "请用 getSchema 确认列名后重试" | "建议改用 DuckDB SQL 或简化分析逻辑" |

**设计细节**：`retryOn` 谓词判断是否需重试。SQLException 由 duckdbRetry 处理，非 SQLException（如 Python 执行异常）穿透到 sandboxRetry。`RETURN_MESSAGE` 将错误格式化为 JSON 返回给 LLM 而非抛异常中断流程，支持 Agent 自愈。

## Q10：SSE 流式输出了哪些事件类型？为什么设计多种事件？

**五种事件类型**：

| event 类型 | 用途 | 触发时机 |
|-----------|------|---------|
| `message`（默认） | 文本 token | LLM 流式输出 |
| `status` | 阶段状态提示 | "正在分析数据…"、"正在生成图表…" |
| `chart` | ECharts 配置 JSON | Synthesizer 构建完成 |
| `table` | 结构化数据快照 | DuckDbQueryTool 查询结果 |
| `complete` | 完成信号（含结构化事件） | 整轮结束 |

**设计原因**：前端按事件类型分类消费——`status` 显示进度提示、`chart` 直接渲染图表、`table` 展示交互式数据表格。单一文本流无法满足这种多模态消费需求。前端 fetch + ReadableStream 解析 SSE 协议，按 `\n\n` 分割事件。

## Q11：怎么评估 Agent 的执行效果？有监控吗？

**Token 精确计量**：`TokenRecordingChatModel` 拦截器拦截 `call/stream` 方法的 response，从中提取 `Usage.getPromptTokens()` + `getCompletionTokens()`，按 `conversationId` 记录到 `TokenLedger`。`TokenLedger` 累计每轮每个模型的消耗。

**步骤追踪**：`AnalysisState.addStepResult` 记录每步工具名称、outputKey、行数、状态（SUCCESS/FAILED）、错误信息。`toContextString()` 生成的摘要包含执行步骤状态，可用于评估 Agent 行为。

**没有外部监控**：当前无 Prometheus/Grafana 等外部监控集成。仅通过日志（SLF4J + rector.netty TRACE）排查问题。

---

## Q12：对话续接（Continue）怎么实现的？

基于 `RunnableConfig.threadId = conversationId` + `RedisSaver` Checkpoint。每轮 Agent 执行完成后，框架自动将完整图状态（含对话历史、Agent 中间状态）Checkpoint 到 Redis。恢复时：

1. 前端发起 Resume 请求（含 conversationId）
2. AgentServiceImpl 从 Redis 读取 Checkpoint
3. 以相同 threadId 启动新轮次，框架自动恢复上一轮状态
4. `AnalysisState.restore()` 恢复文件列表、步骤记录等结构化数据（**不恢复 intent**，每轮重新声明）
5. SummarizationHook 检查是否需要压缩历史

## Q13：为什么在 AnalysisState 里 intentCategory 只用普通字段而不是 Map？

**设计疏忽**。其他状态（loadedFiles、stepResults、dataIndex）都是用 `ConcurrentHashMap<String, ...>` 按 threadId 隔离的，但 intentCategory 是普通实例字段。单用户场景下每次只有一个对话，每轮 clear() 重置，所以没出问题。

**如果修复**：应该改为 `ConcurrentHashMap<String, String> intentCategoryMap` 按 threadId 隔离，与其余状态保持一致。当前状态下并发操作存在潜在的交叉风险。

## Q14：LLM API Key 是怎么保护的不被泄露？

| 环节 | 保护措施 |
|------|---------|
| 前端输入 | 明文输入到浏览器→API 传输（HTTPS） |
| 服务端接收 | 立即调用 `AesGcmUtil.encrypt()` AES-GCM 加密 |
| 存储 | 密文存入 Redis（key: `model:config:default`） |
| 日志 | `ModelProperties.apiKey` 标注 `@ToString.Exclude`，日志打印时自动省略 |
| 运行使用 | `ModelManager` 从 Redis 读取 → `AesGcmUtil.decrypt()` 解密 → 注入 LLM SDK |
| 加密密钥 | 来自 `model.encryption.key` 配置（开发环境有默认值，生产需环境变量），SHA-256 衍生为 256 位 AES 密钥 |

**加密格式**：每次加密生成随机 12 字节 IV，密文附加 IV 前缀后 Base64 编码。反序列化时无法反推出原始 key。

## Q15：文件清理怎么做的？防止误删在线文件？

`FileCleanupJob` 定时任务（cron: `0 0 3 * * ?` 每日凌晨 3 点）。

**清理逻辑**：
1. 扫描存储目录下所有 Parquet 文件
2. 查询 H2 data_file 表中 status=READY 的 viewName 列表
3. 不在此列表中的 Parquet → 标记为孤儿文件
4. 检查文件最后修改时间，超过 `stuck-hours=1` 小时才删除（保护正在写入的文件）
5. 删除孤儿 Parquet

---

## Q16：怎么保证多用户（如果未来扩展）场景下的数据隔离？

当前是单用户应用，使用硬编码默认用户。扩展为多用户的设计预留：

1. **H2 表**：已有 `userId` 字段在 data_file 表，`conversation` 无 userId（需要补充）
2. **AnalysisState**：按 threadId 隔离，扩展为 `user:conversation` 复合 key
3. **文件存储**：当前按日期分目录，可扩展为 `{storageDir}/{userId}/{dateDir}/` 层级
4. **API**：前端请求已带 `credentials: 'include'`，可扩展为 session/cookie 鉴权
5. **ModelManager**：模型配置按用户隔离（当前是单 key `model:config:default`）

## Q17：图表构建抛出异常了怎么办？有兜底机制吗？

两层兜底：

1. **代码级重试**：`buildChart` 方法内捕获异常后自动重试一次（`switch` 重新分发到对应图表构建器）
2. **工具级兜底**：`ChartOutputTool` 抛出的未被捕获异常 → `toolErrorHandler` 拦截 → 返回 `{"error":"system","message":"图表构建失败: xxx"}` → LLM 收到后可修正参数（如换图表类型、调整数据引用）重试

异常信息同时记录到 `AnalysisState.addStepResultFailed()`，注入到下轮上下文中，辅助 LLM 修正。

---

## Q18：多轮对话中的 "失忆" 问题怎么解决的？

**三级递进方案**：

| 级别 | 问题 | 方案 | 效果 |
|------|------|------|------|
| 工作记忆 | 本轮忘记文件结构 | `AnalysisState.toContextString()` 每轮注入 ~200 token 摘要 | ✅ 文件列表和步骤结果始终在上下文中 |
| 短期记忆 | 几轮后历史过长 | `SummarizationHook` 20k token 阈值自动摘要 | ✅ 超限后压缩，不断上下文 |
| 长期记忆 | 对话中断后恢复 | `RedisSaver` Checkpoint + `restore()` 恢复文件/步骤 | ✅ 但意图每轮重新声明 |

**核心优化**：结构化的文件信息（viewName、列名、列类型、行数）直接注入 System Prompt 而非藏在历史消息里，LLM 不会"忘了有什么表"。

---

## Q19：为什么要用 Caffeine 缓存 SQL 结果？和 Redis 缓存怎么分工？

**Caffeine（本地缓存）**：
- 缓存 DuckDbQueryTool 的 SQL 查询结果
- TTL 60 秒，最大 100 条，SHA256(sql + viewNames) 建 key
- 场景：同一轮中 Agent 多次调 getSchema 或重复查询
- 理由：低延迟（0ms），不需要网络

**Redisson（分布式缓存）**：
- 缓存 ToolCacheManager 管理的工具结果（如 validateChart）
- RMapCache，按 `tool:toolName:argsHash` 建 key
- 场景：跨请求复用确定性工具的校验结果
- 理由：本地重启后缓存不丢

**选择逻辑**：高频小数据（SQL 查询结果）用 Caffeine 本地缓存避免网络开销；低频校验结果跨请求复用用 Redis 分布式缓存持久化。

## Q20：如果你现在要重构这个项目，你会改什么？

1. **intentCategory 按 threadId 隔离**：改为 `ConcurrentHashMap`，消除并发场景的交叉风险
2. **拦截器按工具名过滤**：`ToolRetryInterceptor.toolNames` 参数为空，当前所有拦截器应用到全部工具。应设置只对指定工具生效
3. **引入框架级的工具调用排序**：当前意图守卫是 prompt + 错误信号软约束，应在 ReactAgent 工具执行前插入拦截器强制要求 declareIntent 先调用
4. **SummarizationHook 阈值配置化**：当前硬编码 20k，应移至 application.yml
5. **新增监控集成**：接入 Micrometer + Prometheus，暴露 Agent 执行步骤数、工具调用次数、成功/失败率等指标
6. **删除 Prompt 中的中文**：当前 prompt 含中文注释和指令，如果模型切换为英文模型会失效。应抽取为双语或纯英文
