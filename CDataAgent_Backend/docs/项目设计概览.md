## 1. Agent 编排架构

### 设计思路

AI 数据分析场景下，LLM 直接调用工具的"平铺 Agent"模式存在三个问题：
1. **工具多导致决策分散**：14 个工具平铺时，LLM 在每一步都需要判断"该用哪个工具"，容易选错或重复尝试
2. **责任不分层**：数据加载、SQL 查询、图表构建混在一起，LLM 可能在查询数据途中突然尝试生成图表
3. **上下文污染**：数据分析的结果细节与最终输出混杂在同一个 LLM 上下文中

核心思路：**按职责拆分 Agent，每层持有裁剪后的工具列表，减少 LLM 的决策空间**。

### 设计方案

```
ExecutorAgent（数据层）
  持有：DataLoadingTool, DuckDbQueryTool, PythonRunnerTool, PreferenceTool
  职责：自主理解用户需求 → 声明意图 → 加载数据 → 执行分析查询 → 输出文本结论
  检测到需要图表时，在流中输出 ##NEEDS_CHART## 标记

SynthesizerAgent（输出层）
  持有：ChartOutputTool, PreferenceTool
  职责：基于 AnalysisState 中的分析结果 → 调用 buildChart 构建 ECharts JSON → validateChart 校验
```

- 每层是独立的 `ReactAgent` 实例，有独立的 System Prompt
- 数据通过 `AnalysisState` 传递（outputKey → dataJson），而非通过 LLM 上下文
- 超时配置：Executor 90s，Synthesizer 30s，全局 300s

### 技术选型对比

| 方案 | 优点 | 缺点 | 本项目选择 |
|------|------|------|-----------|
| 单 ReactAgent + 14 工具 | 架构简单 | 决策空间过大，LLM 容易误选工具 | ❌ |
| **双层 ReactAgent** | 职责清晰，每层决策空间小 | 串行延迟累加 | ✅ Executor + Synthesizer |
| 三层 ReactAgent（Planner） | 意图推理更独立 | 增加延迟，Planner 对 DeepSeek 推理要求高 | ❌ 已移除 |
| 多 Agent 并行编排 | 响应快 | 协调复杂，结果冲突风险高 | ❌ 不适合分析场景 |

## 2. 上下文与记忆管理

### 设计思路

数据分析是典型的长对话场景（用户持续追问、换维度、换指标看数据），LLM 上下文窗口有限。需要**三级记忆体系**，按粒度和生命周期分层：

1. **工作记忆**：本轮分析的结构化状态（文件、步骤、中间结果）— 以摘要注入，而非全文
2. **短期记忆**：最近几轮对话历史 — 超过阈值自动压缩
3. **长期记忆**：跨对话的持久化状态 — Redis 存储，支持中断恢复

### 设计方案

```
┌─ 第一层：AnalysisState（结构化工作记忆）───────┐
│  维护：文件列表、步骤记录、数据索引、意图声明     │
│  每轮开始 → 从 Redis 恢复结构化数据              │
│           → toContextString() 生成 ~200 token   │
│           → 注入到 System Prompt 尾部            │
│  每轮结束 → clear() → 持久化到 Redis（3天 TTL） │
└─────────────────────────────────────────────────┘

┌─ 第二层：SummarizationHook（上下文压缩）─────────┐
│  监听 Token 消耗，超 20k 阈值时触发              │
│  压缩策略：摘要旧消息 + 保留前 6 条关键消息      │
│  保留首条用户消息（keepFirstUserMessage=true）   │
└─────────────────────────────────────────────────┘

┌─ 第三层：RedisSaver Checkpoint（对话持久化）─────┐
│  基于 RunnableConfig.threadId = conversationId   │
│  每轮执行后自动 Checkpoint                       │
│  TTL: 3 天，过期自动清理（TtlRedisSaver 包装）   │
│  恢复：threadId + REST API → resume 续接          │
└─────────────────────────────────────────────────┘
```

### 技术选型对比

| 功能 | 方案 | 对比 |
|------|------|------|
| 工作记忆 | **AnalysisState**（ConcurrentHashMap） | VS 全量注入上下文：摘要方式节省 80%+ token |
| 上下文压缩 | **SummarizationHook**（20k 阈值） | VS 固定窗口截断：智能摘要保留语义 VS 简单截断丢失信息 |
| 对话持久化 | **RedisSaver + TtlRedisSaver** | VS 数据库直存：Redis 性能更高，TTL 自动过期 VS MySQL 需手动清理 |
| Token 计量 | **TokenRecordingChatModel**（拦截器） | VS 估算：精确计数 VS API 返回估算，误差 < 5% |
| 本地缓存 | **Caffeine**（SQL 结果缓存 60s） | VS 无缓存：重复查询（如 getSchema）直接返回 |

## 3. 分析引擎

### 设计思路

数据分析平台的核心需求：**用户上传 Excel/CSV → 快速查询分析 → 支持多文件关联**。需要两个存储角色——**元数据注册中心**和**列式分析引擎**。选择嵌入式方案避免外部依赖，降低部署成本。

### 设计方案

```
H2（元数据注册中心）
  表：conversation / conversation_message / data_file
  存储：文件元数据（文件名、行数、状态）、列 Schema 缓存（columnMeta JSON）
  列 Schema 缓存：列名、类型、采样值 → Agent getSchema 时直接反序列化，不回查 DuckDB

DuckDB（列式分析引擎）
  模式：每个查询创建独立 in-memory 实例，用完即销毁
  数据注册：CREATE OR REPLACE VIEW viewName AS SELECT * FROM 'parquet_path'
  多文件：注册多个 VIEW → SQL 中通过 viewName 引用 → 支持跨文件 JOIN
  资源控制：threads=4, memory_limit=1GB, timeout=30s, max_rows=1000

协作流程：
  1. 文件上传 → H2 写入 data_file 记录（含 columnMeta JSON）
  2. Agent 需要查询 → 从 H2 查 viewName + parquetPath
  3. DuckDbConfig.createConnection(files) → 注册所有 Parquet 为 VIEW
  4. Agent 执行 SQL → DuckDB 返回 JSON 数组
  5. Agent 调 getSchema → 优先从 H2 columnMeta 重建，回退 DuckDB DESCRIBE
```

### 技术选型对比

| 维度 | DuckDB | SQLite | H2 | MySQL |
|------|--------|--------|-----|-------|
| 部署 | 嵌入式 | 嵌入式 | 嵌入式 | 需独立部署 |
| 存储格式 | 列存（Parquet） | 行存 | 行存 | 行存 |
| 分析能力 | ✅ 列式聚合极快 | ❌ 行存，大表聚合慢 | ❌ 行存 | ⚠️ 一般 |
| 文件格式 | 直接读 CSV/Parquet | 需导入 | 需导入 | 需导入 |
| 适用场景 | OLAP 分析 | OLTP 事务 | OLTP 事务 | OLTP 事务 |
| **本项目角色** | **分析计算** | ❌ | **元数据注册** | ❌ |

**为什么选 H2 而非 MySQL**：单用户场景，零安装，文件级数据库，Spring Boot 原生支持。生产环境中如需要可替换为 MySQL（配置变更即可）。

## 4. 意图识别

### 设计思路

用户输入可以归纳为三类：问候/闲聊可直接回复、模糊需求需追问、明确分析需求要进入分析流程。让 LLM 每次都用完整推理来判断意图成本高且不可控，因此设计**双层过滤**——先用零成本的规则处理可枚举场景，再让 LLM 结构化声明非枚举场景。

### 设计方案

```
输入消息
  │
  ├─ Layer 1: IntentRuleMatcher（正则规则）
  │   ├─ GREETING      → 返回问候语
  │   ├─ FAREWELL      → 返回告别语
  │   ├─ SELF_INTRO    → 自我介绍
  │   ├─ FILE_QUERY    → 查询文件列表
  │   ├─ MODEL_QUERY   → 返回模型信息
  │   ├─ SYSTEM_BOUNDARY → 拒绝透露系统信息
  │   └─ VAGUE         → 反问引导
  │     匹配 → 直接返回，零 LLM 调用
  │     未匹配 → 进入 LLM Agent
  │
  └─ Layer 2: Executor Agent
      └─ declareIntent 工具（由 Prompt 引导 + 守卫保障）
          ├─ category=analysis  → 进入数据分析流程
          ├─ category=chitchat  → 直接聊天
          └─ category=vague     → 反问用户
```

**意图守卫机制**：DataLoadingTool 和 DuckDbQueryTool 执行前调用 `checkIntentGuard()`：
- `intentCategory == null` → `"请先调用 declareIntent 声明意图"`
- `intentCategory == chitchat/vague` → `"当前非分析模式"`
- `intentCategory == analysis` → 放行

**注意**：守卫返回的是**错误信号给 LLM**，LLM 可以据此自我修正重新声明。这不是框架级的代码强制。

### 技术选型对比

| 方案 | 延迟 | 成本 | 覆盖场景 | 可控性 |
|------|------|------|---------|--------|
| **规则 + LLM（本项目）** | 规则层 0ms | 规则层免费 | 可枚举 + 无限 | ✅ 精确控制 |
| 纯 LLM 推理 | 1-3s | 每次计费 | 无限 | ❌ 不可控 |
| 纯规则 | 0ms | 免费 | 仅可枚举 | ✅ 但漏过率高 |
| 分类模型（BERT 等） | 50-200ms | 需部署 | 有限类别 | ✅ 需标注数据 |

## 5. 工具系统

### 设计思路

Agent 的能力边界由工具定义。好的工具系统需要解决：**调用错误怎么处理？工具之间如何隔离？返回值如何被 LLM 理解？**

### 设计方案

```
5 个 @Tool 工具：

Executor 侧：
  ├─ DataLoadingTool     — declareIntent + loadData（受意图守卫保护）
  ├─ DuckDbQueryTool     — runDuckdb + queryStatistics + getSchema（受意图守卫保护）
  ├─ PythonRunnerTool    — runPython（可选 Docker 沙箱）
  └─ PreferenceTool      — getPreference + setPreference

Synthesizer 侧：
  ├─ ChartOutputTool     — buildChart（基于 echarts-java 确定性构建）
  └─ PreferenceTool      — getPreference
```

**结构化错误协议**：

```json
// 语法错误（LLM 可修正）
{"error":"syntax","message":"列名 'sales' 不存在，请用 getSchema 确认后重试"}

// 超时（需简化查询）
{"error":"timeout","message":"查询超时，建议添加 WHERE 筛选或分步查询"}

// 系统异常（可重试）
{"error":"system","message":"数据引擎异常，请重试"}
```

**拦截器链**（Chain of Responsibility）：

```
Agent 发起工具调用
  │
  ├→ duckdbRetry（ToolRetryInterceptor）
  │   maxRetries=1, initialDelay=500ms, backoffFactor=2.0, jitter=true
  │   retryOn=SQLException.class
  │   onFailure=RETURN_MESSAGE（格式化错误返回给 LLM）
  │
  ├→ sandboxRetry（ToolRetryInterceptor）
  │   maxRetries=1, initialDelay=1000ms, backoffFactor=2.0, jitter=true
  │   retryOn=Exception.class（通用兜底）
  │   onFailure=RETURN_MESSAGE
  │
  ├→ toolErrorHandler（ToolErrorInterceptor）
  │   捕获所有未处理异常，转为 {"error":"system"} 返回
  │
  └→ 实际 @Tool 方法执行
```

**结果缓存**：
- DuckDbQueryTool 结果：Caffeine 本地缓存（60s TTL，100 条上限，SHA256 key）
- validateChart 结果：Redisson RMapCache 分布式缓存（按参数 hash 建 key）
- ToolCacheManager：统一封装 get/put/buildKey

### 技术选型对比

| 方案 | 错误处理 | 重试策略 | 适用场景 |
|------|---------|---------|---------|
| **拦截器链（本项目）** | 工具级配置，差异化重试 | 有限重试 + 指数退避 + 抖动 | 数据查询（瞬态错误多） |
| try-catch 内联 | 代码耦合，难复用 | 手动实现 | 简单场景 |
| Spring RetryTemplate | 全局统一，粒度粗 | 固定策略 | 非 Agent 场景 |
| Resilience4j | 功能强大，配置复杂 | 多种策略 | 微服务架构 |

## 6. 数据流管理

### 设计思路

从用户上传 Excel/CSV 到 Agent 可查询分析，中间经历**格式转换、编码探测、列式存储、去重复用、生命周期管理**五个环节，需要保证数据不丢失、不重复、及时清理。

### 设计方案

```
上传管道：
  MultipartFile
    → 校验（扩展名 xlsx/xls/csv，大小 ≤10MB）
    → SHA256(filename + bytes) 内容去重
      → 命中 → 复用已有 Parquet + H2 记录，跳过转换
      → 未命中 → 继续以下流程
    → XLSX/XLS：EasyExcel 流式读取 → 临时 CSV
    → CSV：直接处理
    → DuckDB read_csv_auto（UTF-8 → GBK 编码回退）
    → COPY TO Parquet（Snappy 压缩）
    → 清理临时文件
    → DESCRIBE Parquet → 列名/类型/采样值 → 写入 H2 columnMeta JSON
    → 标记 status=READY

批量上传事务保证：
  1. 逐个转换所有文件（暂不写 H2，失败回滚临时文件）
  2. 全部成功后 → 删除该对话旧文件（物理删 Parquet + 逻辑删 H2）
  3. 写入新 H2 记录

存储结构：
  {storageDir}/{yyyyMMdd}/{fileId}.parquet
  按日期分目录，方便按时间范围清理

清理策略：
  定时任务：每日凌晨 3 点（cron: "0 0 3 * * ?"）
  清理对象：状态不为 READY 或对话已删除的孤立的 Parquet 文件
  保护窗口：stuck-hours=1（防止正在写入的文件被误删）
```

### 技术选型对比

| 环节 | 选型 | 替代方案 | 选择理由 |
|------|------|---------|---------|
| Excel 读取 | EasyExcel | Apache POI | 流式读取低内存，适合大文件 |
| 列式存储 | Parquet | CSV / JSON / Arrow | 列存压缩率高，DuckDB 原生支持 |
| 编码检测 | DuckDB read_csv_auto | 第三方库检测 | 利用 DuckDB 内置能力，减少依赖 |
| 内容去重 | SHA256 | MD5 / 文件大小比对 | SHA256 防碰撞，精度高 |
| 缓存 | Caffeine + Redisson | Redis 全量 / 无缓存 | 本地高频 + 分布式跨请求两级缓存 |
