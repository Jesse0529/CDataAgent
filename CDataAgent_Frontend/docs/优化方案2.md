# 前端流式展示与渲染协议优化方案（第二阶段）

## 1. 文档目标与边界

本文定义前端如何消费后端的运行事件，并把它们渲染为稳定、连续、可恢复的数据分析对话体验。

本次优化的目标不是恢复 Agent 原始 token 的逐字透传。原始 token 可能包含 Markdown、JSON、工具参数、推理过程或不完整片段，不能作为用户界面的数据来源。本次只消费后端定义的、可展示的类型化事件。

本文只覆盖前端改造。事件生产、运行隔离、持久化和重放由后端方案负责；前端不得通过猜测文本内容、正则解析 Markdown 或解析模型 JSON 来补偿后端协议缺失。

## 2. 目标体验

用户发送分析请求后，界面应在短时间内显示“已开始”和当前阶段；随后平滑显示服务端确认安全的摘要增量、表格和图表；运行结束时由最终展示文档一次性确认结果。

普通对话和模糊需求不要求存在分析展示计划。服务端会将已净化的自然语言回答封装为 `paragraph` 类型的 `RenderDocument`；前端按普通文档区块显示它，而不是把空计划渲染成失败提示或回退显示原始文本。

页面刷新后，历史消息必须与实时完成态一致。网络重连时，不重复追加文本、不重复插入图表或表格、不串入其他提问。

## 3. 前端消息模型

将现有 `content`、`tables`、`chartOption`、`renderDocument` 并行维护的方式收敛为一个面向展示的消息部件模型。迁移期可以保留旧字段作为兼容投影，但不得再作为新协议的权威状态。

```ts
type MessagePart =
  | { id: string; type: 'progress'; stage: string; label: string; state: 'running' | 'done' | 'failed' }
  | { id: string; type: 'text'; text: string; streaming: boolean }
  | { id: string; type: 'table'; data: TableEventData }
  | { id: string; type: 'chart'; chartIndex: number; option: Record<string, unknown> }
  | { id: string; type: 'notice'; level: 'info' | 'warning'; text: string }

interface AgentRunState {
  runId: string
  lastEventId: string | null
  resumeToken?: string
  status: 'running' | 'completed' | 'failed' | 'cancelled'
  parts: MessagePart[]
  finalDocument?: RenderDocument
}
```

`RenderDocument` 仍是完成态和历史态的权威展示模型。实时 `parts` 是运行中的 UI 状态；收到 `document.commit` 后，前端校验并写入 `finalDocument`，将页面切换为文档渲染。旧客户端或旧历史仅使用后端提供的纯文本兼容投影，前端不自行降级为 Markdown 渲染。

## 4. 事件消费契约

前端 SSE 解析器必须按事件类型分发，不能把未知事件或解析失败的数据传给 token 回调。

| 事件 | 前端操作 | 幂等键 |
|---|---|---|
| `run.started` | 创建/更新 `AgentRunState`，记录 `runId`、令牌和首个 `eventId` | `runId:eventId` |
| `progress.updated` | 按 `partId` 原位更新进度部件 | `runId:eventId` |
| `preview.start/delta/end` | 仅更新 `text` 部件；使用 rAF 合帧 | `runId:eventId` |
| `artifact.upsert` | 按产物 `id` 新增或覆盖表格、图表部件 | `runId:eventId` + artifact id |
| `document.commit` | 校验 `RenderDocument` 后原子提交完成态 | `runId:eventId` |
| `run.completed` | 收尾 loading、时间、用量；不自行伪造最终文档 | `runId:eventId` |
| `run.failed` | 显示可读失败提示及重试入口 | `runId:eventId` |
| `ping` | 忽略 | 无 |

所有事件必须含 `runId`、`eventId`、`type` 和 `payload`。前端只接受当前消息对应 `runId` 的事件；已处理的 `eventId` 必须去重。解析失败、未知类型、缺少身份字段、文档校验失败均记录诊断信息并安全丢弃，绝不追加到 `content`。

## 5. 流畅性与性能策略

1. `preview.delta` 进入每帧最多一次的缓冲刷新，避免每个网络分片触发 Vue 重渲染和滚动。
2. `progress.updated` 仅在状态或展示文案变化时更新；心跳不触发 UI 更新。
3. 图表使用稳定 `artifact id/chartIndex` 原位更新，避免重复初始化 ECharts 实例。
4. 表格采用显示行数上限、虚拟列表或分页策略；不可将无界查询结果直接挂入响应式树。
5. 自动滚动仅在用户仍位于底部附近时执行；用户向上查看历史时保留阅读位置并显示“有新内容”提示。
6. 不使用 `v-html` 渲染任何模型或事件文本。摘要、提示、标题均使用文本节点渲染。

## 6. 重连与恢复

连接中断后，前端以当前 `runId`、`resumeToken` 和最后成功处理的 `lastEventId` 请求续播。`lastEventId` 必须在事件通过校验并应用到状态后才推进。

重连后允许收到重复事件，但禁止重复应用；若服务端明确表示运行已过期或不可恢复，前端保留已展示结果，显示“本次运行连接已中断，可重新执行”，而不是清空消息或重新拼接旧文本。

取消操作仅取消当前 `runId`；取消后忽略该 run 的迟到事件。新一次提问不得复用旧 run 的令牌、事件游标或文本缓冲。

## 7. 历史消息与兼容策略

消息加载逻辑按以下优先级构造视图：

1. `renderVersion === 1` 且 `renderDocument` 通过本地 schema 校验：渲染 `RenderDocument`。
2. 新协议的确定性纯文本投影：以纯文本兼容组件展示。
3. 历史旧消息：沿用既有安全渲染路径，不对其推断结构化部件。

前端 schema 校验只用于防御性验证，不负责修复文档或重新解释模型输出。校验失败时显示安全提示，并保留可观测日志中的 run/message 标识。

## 8. 实施顺序

### 阶段 A：修复当前链路

- 在 `apiPostStream` 的分片与尾缓冲路径完整传递 `meta/document` 回调。
- `document` 事件无论是否有回调都不得落入 token 分支；非法文档不得显示原始 JSON。
- 修复未使用变量和格式问题，使前端静态检查恢复通过。
- 历史消息映射 `renderDocument/renderVersion`。

### 阶段 B：引入运行事件状态机

- 定义统一事件 envelope TypeScript 类型和 reducer。
- 以 `runId + eventId` 驱动去重与恢复。
- 将实时 `content/tables/chartOption` 写入替换为 `MessagePart` 更新；保留兼容投影。

### 阶段 C：体验与容量优化

- 接入 `progress`、`preview` 和 `artifact` 组件。
- 落实 rAF 合帧、滚动锚定、表格容量保护和图表实例复用。
- 删除新协议路径对 Markdown 解析与文本猜测的依赖。

## 9. 验收标准

- 任意 `document` JSON 不会出现在聊天文本区域。
- 发起请求后即时出现阶段进度；服务端提供 `preview.delta` 时可平滑显示安全文本。
- 刷新页面后，已完成消息与实时最终效果一致。
- 分片 SSE、重复事件、乱序/无效事件、断线重连、取消后迟到事件均不导致崩溃、重复或串流。
- 新协议路径不使用 `v-html`，不解析模型 Markdown，不依赖正则提取 JSON。
- 类型检查、静态检查及上述核心行为测试均通过。

## 10. 本轮落地补充：增量文档与无刷新收敛

### 10.1 问题与目标

当前实现把 `RenderDocument` 作为运行结束后的唯一交付物，并在流结束后以 `loadMessages()` 整体替换消息数组。该策略保证了历史一致性，但使摘要、表格和图表入口在视觉上集中出现。

本轮目标是在不暴露模型原始 token、Markdown、工具参数或 JSON 的前提下，使同一条 AI 消息按服务端确认的产物逐步完成。数据库回读只用于容错同步，不能再承担当前会话的主渲染职责。

### 10.2 前端状态约束

1. `document` 事件允许在同一 `runId` 内发送多个有效快照；后到快照覆盖相同 block id，新增 block 以过渡动画插入。
2. 第一份快照可包含摘要、要点和表格，但不得包含未校验完成的图表 option。它是安全的可展示中间态，不是临时文本拼接。
3. 收到 `chart` 事件时只缓存 option 并维持统一的禁用图表入口；不得提前打开预览，也不得切换成另一套按钮或骨架样式。
4. 收到 SSE `complete` 后，当前内存消息原地进入完成态并开放图表预览；不等待、也不依赖全量历史刷新。
5. 后台回读历史消息时，只与当前流式 AI 消息做字段合并（`renderDocument`、`chartOption`、`tokenUsage` 等），必须保留其客户端 id 和响应式对象引用。禁止重新赋值整个 `messages` 数组。

### 10.3 渲染时序

```text
document(摘要 / 要点 / 表格)  →  渐入渲染已有 block
progress(chart)               →  同款图表入口，禁用
chart(option)                 →  保持入口和已有 block，不打开预览
document(最终快照) + complete →  原地合并，开放预览
后台历史回读                  →  仅校正缺失字段，不触发视觉重置
```

`document` 的每个快照均须通过既有 schema 校验。前端不根据内容推测阶段，也不把快照差异降级为原始文本。

### 10.4 验收标准

- 图表分析中，摘要与表格在图表合成、校验完成前已经可见。
- 最终事件到达时，既有摘要和表格不闪烁、不消失、不重新挂载；仅新增或变更的 block 过渡显示。
- 图表入口在生成中、option 到达后和完成后三个时点保持同一布局与视觉体系；只有 `complete` 后可点击。
- 断线续传的重复快照幂等，不重复插入 block；后台历史回读失败不影响当前已呈现结果。
