# 前端输出格式约束优化方案

> 状态：V2 设计基线，待实施。  
> 配套文档：`CDataAgent_Backend/docs/后端输出格式约束优化方案.md`。两份文档共同定义同一份 `RenderDocument v1` 契约；任一侧实施前必须同时阅读。

## 1. 背景与结论

当前 AI 回答主链路是“模型自由 Markdown token → SSE `message` → 前端正则/启发式 Markdown 修复 → `v-html` 渲染”。模型即使已被 Prompt 要求使用固定标题、空行和表格，仍可能因 token 截断、工具调用后的继续输出、模型不遵从或表格转义而给出非规范 Markdown。

**本次方案的结论：新协议不再把 Markdown 当作前后端展示契约。** 前端只接收并渲染经过后端验证的 `RenderDocument v1`；Markdown 解析器仅保留为历史消息兼容层，不能再承担新消息的正确性保障。

这不是“换一个 Markdown 库”或“增加 Prompt 规则”的需求。格式正确性由类型契约、后端校验、前端受控组件三层保证，Prompt 只影响内容质量。

**V2 补充结论：格式协议改造不能建立在共享可变会话状态和单机 SSE 缓冲之上。** 新协议以一次 Agent 执行为边界；后端生成的 `runId`、严格递增的 SSE 事件 ID 和不可变的最终文档快照共同组成恢复契约。前端绝不能用消息字数、token 数量或 `conversationId` 单独推断恢复位置。

## 2. 现状取证

已确认的当前实现：

- `src/composables/useAgentStream.ts` 将 SSE `message` token 直接拼入 `ChatMessageVO.content`；`complete.analysis` 再覆盖最终内容。
- `src/components/analysis/ChatMessage.vue` 与 `src/utils/messageRenderer.ts` 对文本、标题、表格和流式 token 做手工解析后，通过 `v-html` 展示。
- `src/services/types.ts` 已有 `TableEventData`、`chartOption`、`conclusion` 等部分结构化字段，但尚未形成一份完整的答案文档契约。
- 当前图表弹窗、文件、会话、普通错误提示不属于本次重写对象。

因此，前端目前既是展示层，又在修复上游协议错误；这是后续必须消除的职责倒置。

## 3. 目标、范围与非目标

### 3.1 目标

1. 新协议下，用户可见回答不依赖 Markdown 语法是否完整。
2. 前端不把模型字符串直接放入 `v-html`；文本作为 Vue 文本节点渲染。
3. 表格、图表、结论、列表由显式区块类型表示，前端按区块组件渲染。
4. 新旧消息、旧后端与新后端可在灰度期双向兼容。
5. 新结构可持久化并在刷新、历史加载和断线恢复后保持一致。

### 3.2 实施前置条件

以下能力由后端先提供，前端在未收到 `meta` 确认前不得启用 v1 渲染：

1. 每次请求都对应唯一 `runId`；同一对话的并发执行由后端拒绝或显式排队，不能让两个运行共享临时状态。
2. 后端在本次运行内保存可信工具产物和展示计划，并在 `document` 发出后保存不可变快照；前端不持有可被后续请求篡改的“当前图表/当前表格”全局状态。
3. 所有可恢复事件都有单调递增且运行内唯一的 SSE `id`。恢复请求以 `runId + lastEventId + resumeToken` 定位，客户端按 ID 幂等去重。
4. 部署目标若包含多实例或进程重启恢复，事件缓冲必须由 Redis Stream（或等价持久事件存储）实现；仅 JVM 内存缓冲只能作为单实例开发模式能力，不能标注为高可用。

### 3.3 非目标

- 不在本次替换既有图表算法、文件上传、权限体系或会话业务。
- 不删除历史 `content`、`chartOption` 数据，也不做破坏性数据库迁移。
- 不以“更多正则修复 Markdown”为实施方案；仅保留它服务历史记录。
- 不假定所有模型供应商都支持原生 JSON Schema 输出。

## 4. 目标前端模型

后端传输的文档采用版本化、可判别联合类型。以下 TypeScript 是契约示意；字段名、上限和错误码必须以后端文档为准。

```ts
export interface RenderDocument {
  version: 1
  runId: string
  blocks: RenderBlock[]
  degraded?: boolean
}

export type RenderBlock =
  | { id: string; type: 'summary'; title?: string; text: string }
  | { id: string; type: 'paragraph'; text: string }
  | { id: string; type: 'bullets'; items: string[] }
  | {
      id: string
      type: 'table'
      title?: string
      headers: string[]
      rows: Array<Record<string, string | number | null>>
      totalRows: number
    }
  | { id: string; type: 'chart'; chartIndex: number; title?: string }
  | { id: string; type: 'notice'; level: 'info' | 'warning'; text: string }
```

规则：

- `text` 和 `items` 一律按纯文本显示，不解释 Markdown 或 HTML。后端的 `PlainTextPolicy` 必须在组装前拒绝展示性 Markdown（标题、围栏、表格等）并进行一次受控修复；修复仍失败时以确定性降级文档替代原文，不能把 `##`、断裂表格或 `**` 原样作为正常答案展示。
- `table` 只能由可信工具结果解析得到，不信任模型自行填写的行列数据。
- `chartIndex` 引用同一消息的受控 `chartOption` 数组；越界或不存在时只显示非阻断提示。
- 前端必须对 `unknown` 响应做运行时类型守卫，拒绝格式非法、版本未知或区块字段不完整的文档。

## 5. SSE 与历史消息的前端契约

### 5.1 协议协商

新前端在 `/agent/chat/stream` 和 `/agent/chat/resume` 请求中发送（参数便于代理、日志和故障排查，请求头用于显式能力协商）：

```http
X-Agent-Render-Protocol: render-document.v1
```

```text
renderProtocol=render-document.v1
```

后端首个 `meta` 事件必须回传已协商的协议版本、`runId` 和恢复能力；前端只有在 `renderProtocol === 'render-document.v1'` 时才进入新路径。没有请求头/参数的客户端完整保留现有 `message/status/chart/table/complete` 行为。协商失败或版本未知时，前端必须显式走 legacy 路径，不能假设服务端支持 v1。

### 5.2 新事件

新协议的用户可见正文只来自下列事件：

```text
id: 1
event: meta         // { runId, renderProtocol, resumeToken }
id: 2
event: status       // JSON 进度，不写入最终 document
event: table        // JSON 工具数据预览，不作为 Markdown
event: chart        // JSON 图表配置
event: document     // 已校验的 RenderDocument v1，最终权威答案
event: complete     // JSON：runId、conversationId、tokenUsage、documentVersion
```

`document` 到达前，界面只显示状态、可信表格/图表的加载态，不拼接 Executor 的自由文本 token。`document` 到达后以它原子替换临时状态；`complete` 不是正文的第二来源。

除心跳外，每个事件的 `data` 都是 JSON；新协议不得沿用 `resumeToken:...` 这类文本编码。前端在成功处理事件后才提交 `lastEventId`，以 `(runId, eventId)` 为去重键。恢复请求传递 `runId`、`lastEventId` 和 `resumeToken`，不得继续使用 token 数量或 `resumeSeq`。重放与实时订阅可能短暂重叠，前端必须忽略已处理的 ID，而不是再次追加区块。

### 5.3 历史读取

`MessageVO` 和本地 `ChatMessageVO` 增加可选字段：

```ts
renderDocument?: RenderDocument | null
renderVersion?: number | null
```

渲染优先级固定为：

1. 版本受支持、`runId` 与消息运行一致且运行时校验通过的 `renderDocument`；
2. 旧消息的 `content` + 现有 Markdown 兼容渲染；
3. 安全的纯文本错误提示。

新协议的消息同时保留 `content`，用于旧客户端、搜索和灾难回退；它必须是后端从 `RenderDocument` 确定性生成的兼容投影，不能是模型原始 Markdown，也不是新前端的正常渲染源。

## 6. 组件与代码改造边界

| 位置 | 计划改动 | 不应做的事 |
|---|---|---|
| `src/services/types.ts` | 定义 `RenderDocument`、区块类型、SSE `document` 事件元数据 | 用 `any` 绕过版本和字段检查 |
| `src/services/api.ts` | 解析 SSE `id`、`event: meta/document`，支持协议请求头和参数 | 将 `document` 当普通 token 传给文本回调 |
| `src/composables/useAgentStream.ts` | 区分 `meta`、进度、临时表格、图表与最终文档；保存 `runId/lastEventId` 并按 SSE ID 重连、幂等去重 | 收到 `message` 后继续累计为最终正文，或以 token 计数作为恢复序号 |
| `src/components/analysis/ChatMessage.vue` | 按 `RenderBlock` 分发受控组件，处理加载与降级状态 | 对 `RenderDocument` 使用 `v-html` |
| `src/components/analysis/blocks/*`（新增） | `SummaryBlock`、`BulletListBlock`、`DataTableBlock`、`ChartBlock`、`NoticeBlock` | 将所有区块继续堆回 `ChatMessage.vue` |
| `src/utils/messageRenderer.ts` | 标记为 legacy，仅服务没有 `renderDocument` 的历史数据 | 继续扩展新消息的 Markdown 修复规则 |
| `AnalysisPage.vue` | 只处理会话状态和持久化触发 | 重新承载格式解析或 SSE 细节 |

建议新增一个小型 `renderDocument.ts`：只放运行时守卫、版本判断和安全的 DTO 转换，不能混入 UI 或网络请求。

## 7. 实施顺序与发布策略

1. **阶段 0，后端运行隔离**：先完成 `runId`、单对话并发策略、运行级临时状态和带 ID 的可恢复事件；这一阶段不切换前端渲染。
2. **先后端兼容发布**：后端完成 `renderDocument` 持久化、确定性 `content` 投影和协议协商；默认仍走旧协议。
3. **前端类型与组件**：接入 `RenderDocument`、`meta`、SSE ID 去重与恢复，但保留 legacy 回退；此阶段不删除 `messageRenderer`。
4. **新协议开关**：前端请求参数和请求头同时声明能力，仅内部/灰度用户使用 `render-document.v1`。
5. **观测与修复**：观察文档校验失败、受控降级、未知区块、事件间隙、重放去重和恢复失败。
6. **扩大灰度**：达到验收阈值后默认新协议；旧协议保留至少一个兼容发布周期。
7. **清理**：仅当历史读取和兼容周期满足要求后，收缩新消息路径中的 Markdown 处理。

前端不能先于支持该请求头的后端上线；后端可先上线，因为未协商客户端仍走旧链路。

## 8. 验收标准

- 新协议回答中的标题、列表、表格、图表不依赖 Markdown 解析；故意返回 `## 标题`、未闭合 `**`、断裂 `|` 时，界面仍以安全纯文本或结构化区块正确显示。
- 所有 `document` 在前端通过运行时守卫；非法版本/非法区块不会导致 Vue 渲染异常。
- 新前端加载含 `renderDocument` 的历史记录后，不调用 Markdown 主渲染路径。
- 旧消息、旧前端、无图表回答、纯聊天回答、错误回答均可用。
- 断线重连后，前端以 SSE `id` 而非 token 数量恢复，不重复拼接最终答案。
- 同一对话重复发送、重连回放与实时事件重叠时，前端不会混入另一轮的表格、图表或文档；`runId` 不一致的事件必须丢弃并记录诊断信息。
- `ChatMessage.vue` 不再承担新消息的 Markdown 解析；新 block 组件存在独立单测。
- `npm run check` 通过；另需补充组件测试和模拟 SSE 集成测试。

## 9. 风险与决策约束

- 不能只让模型“输出 JSON”然后直接 `JSON.parse`：模型、网关和流式截断都会产生无效 JSON，必须依赖后端验证和兜底。
- 不能只在前端做 JSON 修复：前端无权校验表格/图表是否真的来自本轮可信工具结果。
- 不能将现有单例 `AnalysisState` 中的 `currentThreadId`、图表列表或 Future 直接作为新协议临时状态；它们在并发运行下会串扰，必须由后端按 `runId` 隔离。
- 不能把 JVM 内存事件环当作多实例高可用的恢复方案；需按部署目标选择 Redis Stream（推荐，项目已有 Redis）或等价持久事件存储。
- 不能立即删除 `content`：历史消息和旧客户端需要兼容。
- 原生 JSON Schema 输出的供应商支持度不是本方案的前提；以已在项目中使用的 Tool Calling 为第一版主通道。
- 本文档只授权格式协议相关改动。若涉及认证、模型切换、表结构清理或大规模 UI 重设计，需单独评审。
